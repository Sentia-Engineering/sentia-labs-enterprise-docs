Sentia uses **WorkOS AuthKit** for enterprise authentication and provisioning. Every workspace is corporate-IdP-only тАФ there are no password flows.

For the user-facing sign-in flow, see [Sign in with SSO](/onboarding/sso-setup). This page is for workspace owners and IT/SecOps administrators.

## What WorkOS provides

| Feature | What it covers |
| --- | --- |
| **SAML 2.0 SSO** | Most enterprise IdPs (Okta, Entra ID, OneLogin, PingFederate, Auth0, Google Workspace, JumpCloud) |
| **OIDC SSO** | Modern OIDC-compliant IdPs (Google Workspace OIDC, Azure AD OIDC, custom OIDC) |
| **SCIM 2.0** | Real-time user/group provisioning from your IdP |
| **Directory Sync** | Read-only sync of users, groups, and group memberships |
| **Magic Link fallback** | Time-boxed magic-link auth for invited users before SCIM is wired up |

## Initial setup

Setup is a guided process with your CSM. The high-level flow:

<Steps>
  <Step title="Pick your IdP and protocol">
    SAML or OIDC. SCIM as a separate configuration.
  </Step>
  <Step title="Get WorkOS metadata">
    WorkOS provides:
    - **ACS URL** (where your IdP posts assertions)
    - **Entity ID** / **Audience URI**
    - **SP signing certificate**
    - For OIDC: client_id, client_secret, redirect URIs
  </Step>
  <Step title="Configure your IdP">
    Your IT team adds Sentia as a SAML/OIDC application. Required attribute mappings:
    - `nameID` or `sub` тАФ user's unique identifier
    - `email` тАФ primary email
    - `firstName`, `lastName`
    - (Optional) `groups` тАФ for role mapping
  </Step>
  <Step title="Test with a pilot user">
    Your CSM verifies a sign-in from a test user.
  </Step>
  <Step title="Roll out">
    Assign all relevant users to the Sentia app in your IdP.
  </Step>
</Steps>

## SCIM provisioning

Strongly recommended. Real-time, automatic, eliminates manual invites and deprovisioning lag.

### What SCIM does

| Event in IdP | Effect in Sentia |
| --- | --- |
| User assigned to Sentia app | User auto-created in workspace |
| User unassigned | User removed; session terminated |
| User attributes change (name, email) | Synced to Sentia profile |
| User added to a group | Role updated based on group mapping |
| User removed from a group | Role downgraded or access removed |
| Group created/renamed | Mapping updates apply |

SCIM operates in real-time тАФ most IdPs push updates within seconds.

### Setup

<Steps>
  <Step title="Generate a SCIM endpoint + bearer token">
    Your CSM provides these from WorkOS. Each workspace has a unique SCIM endpoint.
  </Step>
  <Step title="Configure SCIM in your IdP">
    In Okta: Applications тЖТ Sentia тЖТ Provisioning тЖТ Configure API integration.
    
    In Entra ID: Enterprise Applications тЖТ Sentia тЖТ Provisioning тЖТ Automatic.
    
    Paste the SCIM endpoint URL and bearer token.
  </Step>
  <Step title="Map attributes">
    Default attribute mappings cover most needs. Custom claim mappings (e.g., for non-standard role attributes) are supported.
  </Step>
  <Step title="Enable provisioning + push groups">
    Push groups for role-mapping (sentia-owners, sentia-admins, etc.).
  </Step>
  <Step title="Initial sync">
    Your IdP runs an initial sync of all assigned users. Sentia auto-creates them with appropriate roles.
  </Step>
</Steps>

### Role mapping via groups

Configure in Settings тЖТ Workspace тЖТ Authentication тЖТ SCIM:

| IdP group | Sentia role |
| --- | --- |
| `sentia-owners` | Owner |
| `sentia-admins` | Admin |
| `sentia-operators` | Operator (default for active practitioners not in any higher group) |
| `sentia-viewers` | Viewer |

Group names are configurable. Users in multiple groups get the highest role.

## Supported IdPs

WorkOS supports all standards-compliant SAML 2.0 and OIDC providers. Tested at scale:

- Okta
- Microsoft Entra ID (formerly Azure AD)
- Google Workspace (SAML and OIDC)
- OneLogin
- JumpCloud
- Auth0
- PingFederate / PingOne
- Duo SSO
- Rippling

For any IdP not on this list: SAML 2.0 + OIDC are open standards; if your IdP supports the standards, it'll work.

## Multi-IdP per workspace

Most workspaces use one IdP. Multi-IdP is supported for:

- Companies in the middle of an IdP migration (Okta тЖТ Entra ID)
- Joint ventures with two parent companies
- M&A integration periods

Multi-IdP setup requires CSM coordination тАФ there are edge cases around overlapping email domains.

## Test before rolling out

WorkOS provides a **test connection** mode where you can:

- Send a SAML assertion through and inspect the parsed response
- Verify attribute mappings before pushing to all users
- Test SCIM CRUD operations

Use this before pushing changes to production sign-in flows.

## Common configurations

<AccordionGroup>
  <Accordion title="Okta + SAML + SCIM">
    1. In Okta, create a new SAML 2.0 Web App with the WorkOS-provided ACS URL and Entity ID.
    2. Map `email`, `firstName`, `lastName`, and (optional) `groups` to standard Okta attributes.
    3. Assign users individually or via group push.
    4. Enable SCIM: Provisioning tab тЖТ Configure API integration тЖТ paste WorkOS SCIM endpoint + token.
    5. Test with a pilot user before pushing to your full org.
  </Accordion>

  <Accordion title="Entra ID + OIDC + SCIM">
    1. Enterprise Applications тЖТ New application тЖТ Sentia (from gallery if listed, otherwise non-gallery).
    2. Single sign-on тЖТ OIDC тЖТ paste WorkOS redirect URI and configure attribute mappings.
    3. Provisioning тЖТ Automatic тЖТ paste WorkOS SCIM endpoint + bearer token.
    4. Push group assignments.
    5. Test sign-in from a pilot user.
  </Accordion>

  <Accordion title="Google Workspace + SAML">
    1. Admin Console тЖТ Apps тЖТ Web and mobile apps тЖТ Add custom SAML app.
    2. Paste WorkOS ACS URL and Entity ID.
    3. Configure attribute mappings.
    4. Assign org units.
    
    Google Workspace doesn't support SCIM directly тАФ use Google Cloud Directory Sync (GCDS) with WorkOS Directory Sync API as a bridge.
  </Accordion>

  <Accordion title="JumpCloud + OIDC + SCIM">
    1. SSO тЖТ +Application тЖТ Custom OIDC App.
    2. Configure redirect URI and scopes (openid, email, profile, offline_access).
    3. Identity Management тЖТ API Connections тЖТ SCIM тЖТ paste WorkOS endpoint + token.
    4. Assign user groups.
  </Accordion>
</AccordionGroup>

## Troubleshooting

<AccordionGroup>
  <Accordion title="SAML assertion arrives but auth fails">
    Most often: `email` attribute is missing or has unexpected case. Check WorkOS logs (your CSM has access) for the exact failure reason.
  </Accordion>

  <Accordion title="SCIM provisioning succeeds in IdP but user can't sign in">
    The user was created in WorkOS but not propagated to Sentia. Force a SCIM resync from your IdP, or contact support to manually re-trigger the WorkOS тЖТ Sentia user push.
  </Accordion>

  <Accordion title="Role mapping isn't working">
    Group push isn't enabled in your IdP, or the group names don't match the mapping configured in Sentia. Verify:
    - Groups are being pushed (check WorkOS Directory Sync logs)
    - Group names exactly match the mapping (case-sensitive)
  </Accordion>

  <Accordion title="Multi-domain support">
    By default, a workspace is mapped to one or more email domains. If your company has multiple domains (acquisitions, regional brands), add them in Settings тЖТ Workspace тЖТ Authentication тЖТ Email domains.
  </Accordion>
</AccordionGroup>

- SCIM CRUD operations (create/update/delete)
- Role changes via SCIM
- Configuration changes
