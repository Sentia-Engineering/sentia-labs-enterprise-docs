Sentia is built for enterprise security review. This page is the at-a-glance security posture — share it with your IT/SecOps team or vendor management process.

## Architecture in one paragraph

All data is workspace-scoped. PostgreSQL row-level security enforces complete isolation between tenants. Authentication is corporate-IdP-only (SAML/OIDC via WorkOS), with no password flows. Every credential is encrypted at rest with per-tenant KMS keys. Every mutation is audit-logged. LLM traffic uses zero-retention DPAs with both Anthropic and Google Vertex. EU customers get full eu-west-1 data residency.

## Authentication & access

- **WorkOS SSO** — SAML 2.0 / OIDC via your corporate IdP
- **SCIM** — real-time user provisioning and deprovisioning from your IdP
- **No passwords** — Sentia stores zero password hashes
- **RBAC** — 4 roles enforced at API middleware; every authorization check audit-logged
- **Sessions** — httpOnly JWT cookies, 12h inactivity timeout, immediate revocation on SCIM deprovisioning
- **API keys** — scoped to workspace, listed and revocable, every usage audit-logged

See [SSO & SCIM](/admin/sso-scim).

## Tenant isolation

- **PostgreSQL row-level security (RLS)** — `FORCE ROW LEVEL SECURITY` applied on every tenant table
- **API workspace scoping** — every request requires an `x-workspace-id` header that's cross-checked against the bearer JWT's allowed workspaces
- **Vector store isolation** — Pinecone namespaces are workspace-scoped; cross-workspace retrieval is impossible
- **Object storage isolation** — S3 prefixes per workspace with bucket policies; no cross-tenant access paths

There's no admin path that operates across workspaces silently. Cross-workspace queries by Sentia staff require explicit support-mode access, which is itself audit-logged and time-limited.

## Encryption

| Where | How |
| --- | --- |
| **In transit** | TLS 1.3 everywhere. HTTPS-only. HSTS enabled. |
| **At rest** | AES-256 across all data stores (PostgreSQL, MongoDB, Redis, Pinecone, S3) |
| **Credentials** | Per-tenant KMS keys (AWS KMS) wrap integration credentials and API tokens |
| **Secrets** | AWS Secrets Manager only. Never in env vars, code, or logs. |
| **Backups** | Encrypted with KMS-managed CMKs; backup keys are separate from primary keys |

Per-tenant KMS means: if a customer requests data destruction, deleting their KMS key cryptographically shreds their data — even backups become unreadable.

## Network & infrastructure

- **AWS-only** — EKS, Aurora PostgreSQL, ElastiCache Redis, Pinecone, MSK Kafka (Redpanda), KMS, GuardDuty
- **Private subnets** — application services in private subnets; only the WAF + ALB are internet-facing
- **VPC peering / PrivateLink** — available for integration endpoints on request
- **DDoS protection** — AWS Shield Standard; Shield Advanced for enterprise tier
- **WAF** — managed rules + custom rules for known abuse patterns
- **SSRF protection** — request-handling code blocks private IPs, cloud metadata endpoints, IPv4-mapped IPv6
- **Egress filtering** — only declared external services (LLM providers, integration APIs) are reachable from production workloads

## LLM safety

- **Zero-retention DPAs** with Anthropic and Google Vertex — no training on customer data
- **No customer data in committed prompts** — Sentia stores prompt templates in code; customer data is injected at runtime only
- **No prompt content in logs** — prompts and completions are not logged; only token counts and aggregate metadata
- **PII redaction** at the integration boundary before verbatims enter the embedding pipeline
- **Dual provider** with circuit breaker — Anthropic primary, Google Vertex fallback; failover preserves privacy guarantees

See [LLM safety](/security/llm-safety).

## Auditing & monitoring

- **Audit log** — every mutation, sign-in, integration action, admin operation. 90 days hot, 1 year warm, 7 years cold.
- **Error monitoring** — Sentry with PII scrubbing
- **Performance monitoring** — Web Vitals, server-side traces; no customer-data telemetry
- **GuardDuty** — AWS-native threat detection on all production accounts
- **Anomaly alerts** — failed-auth surges, unusual API patterns, sudden config changes

See [Audit log](/admin/audit-log).

- **Customer notification** — SEV-1 within 30 minutes via Slack Connect + status page; SEV-2 within 2 hours
